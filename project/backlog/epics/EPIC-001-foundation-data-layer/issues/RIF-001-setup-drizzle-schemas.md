## Problem

Necesitamos 6 schemas Drizzle como SSOT del modelo de datos (per `06_DATA_MODEL.md`). Sin esto, ningún feature puede tocar la DB.

## Acceptance Criteria

```gherkin
Given un repo fresco con starter kit
When agrego los schemas en src/lib/db/schema/{raffles,prizes,sellers,buyers,tickets,admin-actions}.ts
Then pnpm db:generate produce una migration SQL sin errores
And pnpm db:migrate la aplica en Neon dev branch sin errores
And puedo importar { raffles, prizes, sellers, buyers, tickets, adminActions } desde src/lib/db/schema
And los tipos TypeScript inferidos son correctos (typecheck pasa)
```

### Constraints verificables (per BR + doc 06)

```gherkin
Given el schema final
Then la tabla 'tickets' tiene unique constraint sobre (raffle_id, number)
And la tabla 'raffles' tiene unique constraint sobre public_slug
And la tabla 'sellers' tiene unique constraint sobre access_token
And la tabla 'prizes' tiene unique constraint sobre (raffle_id, position)
And FK 'tickets.buyer_id' tiene onDelete: 'set null'
And FK 'tickets.seller_id' tiene onDelete: 'set null'
And FK 'tickets.raffle_id' tiene onDelete: 'cascade'
And el enum 'raffle_status' tiene ['draft', 'open', 'drawn']
And el enum 'ticket_status' tiene ['available', 'sold']
And buyers.name, buyers.phone, buyers.email aceptan NULL (BR-008)
```

## Implementation notes

- Sigue exactamente schemas de `06_DATA_MODEL.md` §E-001 a §E-006
- One file per domain (per SK.md §1.2)
- Index `(status, archived_at, created_at DESC)` en raffles (dashboard query)
- Index `(raffle_id, status)` en tickets (grilla pública query)
- NO usar `pnpm db:push` (SK.md §1.1) — siempre `generate` + `migrate`
- Actualizar `project/reference/INVENTORY.md` post-commit (autogen)
