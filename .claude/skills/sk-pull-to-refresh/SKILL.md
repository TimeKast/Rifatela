# sk-pull-to-refresh — Kit-Shipped Pull-to-Refresh Primitives

> Pair: PTR-side patterns are still kit-specific; if a second project replicates this, split into `kb-pull-to-refresh` for the portable hook contract. For the SW + install + push infra this primitive sits on top of, see [`sk-pwa`](../sk-pwa/SKILL.md).

**What this skill covers:** the two PTR primitives the kit ships and how to choose between them. PTR is **infrastructure, not policy** — the kit gives you the gesture, the indicator, and a sensible default callback (`router.refresh()`); each derived screen decides whether refreshing is meaningful and, when needed, supplies its own callback via the per-screen wrapper.

> **Registry anchors** — canonical import paths in `project/reference/HOOKS.md`; component catalog in `project/reference/INVENTORY.md`; both are autogenerated and the source of truth (don't enumerate exports here, they drift). The kit ships the hook at `src/lib/pwa/usePullToRefresh.ts`, shell context helpers at `src/lib/pwa/shellPullToRefresh.tsx`, and the two components under `src/components/pwa/` (re-exported via the `@/components/pwa` barrel). Active mount in `src/app/(protected)/DashboardShell.tsx`; per-screen example in `src/app/(protected)/notifications/notifications-client.tsx`.

## 1. `usePullToRefresh` hook — `src/lib/pwa/usePullToRefresh.ts`

Low-level driver. Listens to `touch{start,move,end}` on the configured target (defaults to `window`, since the kit shell scrolls on the document — `<main>` is not a scroll container). Computes a damped pull distance with linear resistance and invokes `onRefresh()` once the user releases past `threshold`.

### Signature

```ts
function usePullToRefresh(opts: {
  onRefresh: () => void | Promise<void>;
  enabled?: boolean; // default true
  threshold?: number; // default 80 (px after resistance)
  maxPull?: number; // default 120 (px ceiling)
  resistance?: number; // default 2.5 (linear dampening)
  scrollTargetRef?: RefObject<HTMLElement | null>; // default: window
}): {
  pullDistance: number; // 0..maxPull (current damped pull)
  isPulling: boolean; // user is actively dragging
  isRefreshing: boolean; // onRefresh promise in flight
};
```

### Activation rules

- **Top-only:** the gesture only arms when the scroll target is at the top (`scrollTop <= 1`, with 1px tolerance for iOS bounce).
- **Single-finger only:** multi-touch cancels.
- **Direction lock at 10px:** if the user moves horizontally more than vertically in the first 10px, the gesture is abandoned so horizontal scrolling / swipe-back work normally.
- **`enabled: false`** → no listeners are attached, hook is a noop.

### Why it listens on `window` by default

The kit's `DashboardShell` does not give `<main>` an explicit `overflow-y-auto` or fixed height — content scrolls the document. A hook that polled `containerRef.current.scrollTop` would always see `0` regardless of the real scroll position and trigger the gesture mid-page. Listening on `window` and reading `window.scrollY` matches the real scroll target.

If a derived screen owns its own scroll container (for example a fullscreen modal with `overflow-y-auto`), pass that element via `scrollTargetRef` and the hook switches to reading `el.scrollTop`.

### Why `touchmove` is registered with `{ passive: false }`

`preventDefault()` inside `touchmove` is the **primary defence** against the browser's native rubber-band firing simultaneously with the custom PTR. React event handlers can be installed as passive listeners by default in some configurations, which silently turns `preventDefault()` into a noop. The hook bypasses that by calling `addEventListener('touchmove', handler, { passive: false })` directly.

A complementary CSS belt — `overscroll-behavior-y: contain` on the scroll container — is what stops Android Chrome's full-page native PTR from firing on top of ours. The shell applies this class to its root `<div>`; the per-screen wrapper applies it locally to its own subtree (see §3 for the ladder when it needs to escalate to `<body>` / `<html>`).

## 2. `<PullToRefreshShell>` — shell-wide default — `src/components/pwa/PullToRefreshShell.tsx`

The default PTR for any kit-derived app. Mounted once at the protected shell level (`DashboardShell`). All routes under the shell inherit the gesture without per-screen wiring.

### Properties

- **Standalone.** No props. No children. Layout-invisible to consumers — the indicator is `position: fixed` (viewport-level, not `absolute`) anchored to `env(safe-area-inset-top, 0)` so it lives below the notch / system header.
- **Gate:** `isMobile()` — capability/media-query based (`(pointer: coarse) and (hover: none)`), not UA sniffing. Desktop = noop (no listeners attached).
- **Refresh:** hardcoded `router.refresh()` wrapped in `useTransition`. The promise that resolves the touch handler is held until `isPending` settles, with a defensive timeout (~2000 ms) that fires the resolve if the transition is not observable (e.g. routes that don't suspend) — without it, the indicator could hang forever.
- **Opt-out via context:** screens that own their own refresh callback (because their data lives in client state) call `useDisableShellPTR()` to silence the shell while mounted. Counter-based — multiple concurrent opt-outs compose safely.

### Why shell-wide is the equivalent of per-list nativeApp

Native iOS (`UIRefreshControl`) and Android (`SwipeRefreshLayout`) bind PTR to a scroll container, not a screen. Web equivalents do the same — Twitter, Gmail, and Reddit on mobile web mount PTR shell-wide because their feed is the only scroll container. The kit's `DashboardShell` document-scrolls (`<main>` has no `overflow-y-auto`), so the scroll container is `<html>` / `<body>`. A shell-wide mount is the matching pattern. See §7 for the broader reference.

### When the shell does NOT cover a screen

- The screen owns its own scroll container (modal with `overflow-y-auto`, page with sticky header above an internal scrolling table). The shell's `window`-bound gesture will not arm, and even if it did, refresh wouldn't make sense — the user is at the top of an inner element, not the document.
- The screen's data lives in client state and `router.refresh()` does not update it (the canonical case is `/notifications`).

In both cases, use the per-screen wrapper (§3) and pair it with `useDisableShellPTR()` to silence the shell.

## 3. `<PullToRefresh>` — per-screen wrapper — `src/components/pwa/PullToRefresh.tsx`

Use when the shell-wide default does not apply. The wrapper composes `usePullToRefresh` with a default `isMobile()` gate and an indicator local to the wrapped subtree.

### Minimal usage

```tsx
import { PullToRefresh } from '@/components/pwa';
import { useDisableShellPTR } from '@/lib/pwa/shellPullToRefresh';

export function MyClientPage() {
  useDisableShellPTR(); // silence the shell while this wrapper is in charge
  // ...
  return <PullToRefresh onRefresh={refetchData}>{children}</PullToRefresh>;
}
```

### Props

| Prop              | Default      | Notes                                                                                                                                   |
| ----------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------- |
| `onRefresh`       | required     | Sync or async; the spinner stays in indeterminate state until it resolves                                                               |
| `enabled`         | `isMobile()` | Default is the capability gate. Pass `enabled={isInstalled}` for PWA-only opt-in. `false` disables. `true` overrides (useful in tests). |
| `threshold`       | `80`         | Damped pixels the user must reach to commit                                                                                             |
| `maxPull`         | `120`        | Visual ceiling for the indicator                                                                                                        |
| `resistance`      | `2.5`        | Real-px to damped-px ratio                                                                                                              |
| `scrollTargetRef` | `undefined`  | Pass when the consumer screen has its own scroll container                                                                              |
| `className`       | `undefined`  | Extra classes on the wrapper div                                                                                                        |

### Visual contract

- Indicator is `fixed`-positioned, top-anchored with `env(safe-area-inset-top, 0)` so it lives below the notch / system header without overlapping the kit's sticky `Header`.
- Spinner rotation is proportional to `pullDistance / maxPull` while the user is dragging, then switches to `motion-safe:animate-spin` while `isRefreshing`. Users with `prefers-reduced-motion: reduce` see opacity instead of rotation.
- The wrapper `<div>` carries `overscroll-y-contain` to help contain the native bounce locally. The shell's root `<div>` carries the same class as the primary defence — escalation ladder if Android Chrome native PTR still fires:
  1. Default — class lives on the shell's root `<div>` (current state).
  2. If native Chrome PTR still fires → move to `<body>` via `useEffect` in `DashboardShell` with cleanup.
  3. If still fires → scope to the protected layout via `globals.css` (`.protected-layout html { overscroll-behavior-y: contain; }`) — never global without scope, because login / marketing pages outside the shell still want the platform rubber-band.

## 4. Cuándo usar / no usar

```
Default (shell-wide) — AUTO
✅ Toda página protegida hereda PTR del kit. No hay nada que escribir.
❌ El shell no aplica si la pantalla tiene su propio scroll container o maneja
   refresh con state cliente — usa el wrapper per-screen y opt-out del shell.

Per-screen wrapper — explícito
✅ Modal scrolleable con overflow-y-auto interno → wrapper con scrollTargetRef
✅ Pantalla con sticky header + tabla con scroll interno
✅ Página que fetchea con useState + action calls (router.refresh no actualiza)
❌ Pantallas con formularios sin guardar — un refresh accidental destruiría el draft
❌ Pantallas de admin desktop-only — no se ejecuta (gate isMobile()), pero no
   tiene sentido conceptual; documenta la decisión.
❌ Reemplazo del único mecanismo de refresh — ver §6 a11y
```

## 5. Patrones de callback (la sección estrella, aplica al wrapper)

El shell hace `router.refresh()` por default — sin callback. Las cuatro shapes de abajo aplican cuando estás usando el **wrapper per-screen**.

| Tipo de pantalla                                  | Callback                      | Ejemplo                                                                          |
| ------------------------------------------------- | ----------------------------- | -------------------------------------------------------------------------------- |
| Listado client-side (estado local + fetch action) | Loader local existente        | `<PullToRefresh onRefresh={() => fetchUsers(page)}>`                             |
| Hook con polling (notifications, etc.)            | `refetchX()` del hook         | `<PullToRefresh onRefresh={refetchNotifications}>`                               |
| Página RSC simple                                 | `router.refresh()`            | `const router = useRouter(); <PullToRefresh onRefresh={() => router.refresh()}>` |
| Tabla con `useServerTableState` (URL params)      | `setPage(1)` o re-push de URL | `<PullToRefresh onRefresh={() => setPage(1)}>`                                   |

### Cuándo aplica cada uno

**Listado client-side.** La pantalla ya orquesta su propio fetch con `useTransition` + estado local (ver `notifications-client.tsx` como referencia). El callback de PTR llama al mismo loader pasándole el `page` actual. **No inventes un nuevo loader** — extrae el existente como `useCallback` antes de pasarlo si todavía no lo está.

**Hook con polling.** Hooks como `useNotifications` exponen un `refetchX()` que el polling y el botón "Actualizar" ya consumen. PTR conecta ahí directamente, sin lógica nueva.

**Página RSC simple con state cliente despreciable.** Si la pantalla es un Server Component y no hay state cliente que se pierda, `router.refresh()` invalida la ruta actual y el RSC re-fetcha en el server. Pero en este caso **probablemente no necesitas el wrapper** — el shell hace exactamente lo mismo. Usa el wrapper solo si el RSC tiene un scroll container interno propio.

**Tabla con `useServerTableState`.** El estado vive en URL params (`?page=`, `?sort=`, etc.). El callback más limpio es `setPage(1)` (o equivalente), que re-pushea la URL y dispara el refetch automático.

## 6. Integration checklist

### Default case — nada que hacer

`<PullToRefreshShell />` ya está montado en `DashboardShell`. Cualquier página protegida nueva hereda PTR. La pantalla no necesita imports, hooks ni wrappers.

### Custom scroll container / state cliente — wrapper per-screen

1. **Imports:**
   ```ts
   import { PullToRefresh } from '@/components/pwa';
   import { useDisableShellPTR } from '@/lib/pwa/shellPullToRefresh';
   ```
2. **Llama `useDisableShellPTR()`** al inicio del componente — silencia el shell mientras esta pantalla maneja el gesto. Sin esto, el shell PTR y el wrapper disparan los dos.
3. **Identifica el callback** según §5. Si la pantalla todavía no expone uno reutilizable, extrae el loader existente como `useCallback` — no escribas uno nuevo.
4. **Wrap el subtree de la pantalla**:
   ```tsx
   return (
     <PullToRefresh onRefresh={...}>
       <div className="flex flex-col gap-4">{/* contenido */}</div>
     </PullToRefresh>
   );
   ```
5. **Si la pantalla tiene scroll container interno** (modal, tabla con scroll fijo): pasa `scrollTargetRef={ref}` apuntando al elemento que scrollea, no al wrapper.
6. **Agrega un botón "Actualizar" visible** (icon `RefreshCw` de `lucide-react`) que invoque el mismo callback. Es la a11y rule del §7 — no es opcional.

## 7. Anti-patterns

```
❌ Mountar <PullToRefreshShell> en una pantalla específica
   Es shell-only por diseño — vive en DashboardShell, una sola vez.

❌ Mountar <PullToRefresh> wrapper en pantalla cubierta por el shell SIN llamar useDisableShellPTR()
   Resultado: double-fire del gesto. Patrón canónico: el wrapper se queda + la página
   llama useDisableShellPTR() para silenciar el shell mientras está mounted.

❌ Asumir que router.refresh() refresca state cliente
   Invalida el RSC tree pero NO toca state cliente (useState, fetched-via-action).
   Si la pantalla maneja su lista con setItems(...) tras llamar getX(), router.refresh()
   no actualiza nada visible. Usa wrapper per-screen + callback explícito.

❌ Confiar en el gesto como único mecanismo de refresh
   PTR es gesture-only — usuarios con motor impairments, navegadores sin touch, o
   keyboard-only no pueden invocarlo. Acompaña con un botón "Actualizar"
   (icon + aria-label) en la barra de acciones de la pantalla.

❌ Inventar un nuevo loader cuando la pantalla ya tiene uno
   Si ya hay fetchData(page), refetchX() o equivalente, reusalo. PTR es solo el trigger,
   no el data layer.

❌ Mover overscroll-behavior-y: contain a globals.css sin scope
   Sí escala — pero scoped al protected layout, no global. Globalizar afecta /login
   y otras rutas fuera del shell, donde el rubber-band nativo del browser sigue
   siendo el comportamiento esperado.

❌ Mountar wrapper PTR adentro de un modal scrolleable sin pasar scrollTargetRef
   El hook escuchará window por default y se armará cuando el documento esté en top,
   no cuando el modal lo esté. Pasa el ref del scroll container interno.
```

## 8. Mobile app patterns reference

Justifica la decisión arquitectónica (shell-wide como default):

| Plataforma                          | Pattern              | Razón                                                   |
| ----------------------------------- | -------------------- | ------------------------------------------------------- |
| iOS UIRefreshControl                | Per-scroll-view      | UIKit ata refresh al scroll container.                  |
| Android SwipeRefreshLayout          | Per-scroll-container | Mismo principio.                                        |
| Twitter / Gmail / Reddit mobile web | Shell-wide en feed   | Document-scroll en main view; un solo scroll container. |
| Slack mobile web                    | Per-channel          | Cada canal scrollea independientemente.                 |
| Linear / GitHub / Notion mobile web | Sin PTR              | Realtime sync hace PTR redundante.                      |

**Regla:** PTR vive donde está el scroll container. El kit `DashboardShell` document-scrollea → un solo scroll container raíz → shell-wide es el equivalente correcto al per-scroll-view nativo.
