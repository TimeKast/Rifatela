## 1. Table System

The table system has three layers:

| Layer                    | Entry point                     | Best for                                 |
| ------------------------ | ------------------------------- | ---------------------------------------- |
| **"Batteries included"** | `DataTable`                     | 90% of tables — built-in search + paging |
| **Composable**           | `Table` + `useTableState`       | External filter bars, custom toolbars    |
| **Server-driven**        | `Table` + `useServerTableState` | 100+ rows, URL-synced filters            |

### 1.5 Hooks — `useTableState` vs `useServerTableState`

| Hook                  | Data size | State lives in | Best for                           |
| --------------------- | --------- | -------------- | ---------------------------------- |
| `useTableState`       | <100 rows | Client memory  | Catalogs, inline editing           |
| `useServerTableState` | 100+ rows | URL params     | Logs, transactions, shareable URLs |

```tsx
import { useTableState } from '@/lib/hooks/useTableState';

const { paginatedData, onSort, getSortDirection, setSearch, page, totalPages, setPage } =
  useTableState({
    data: products,
    searchableColumns: ['name', 'sku'],
    pageSize: 20,
  });
```

```tsx
import {
  useServerTableState,
  PAGE_SIZE_OPTIONS,
  DEFAULT_PAGE_SIZE,
  MAX_PAGE_SIZE,
} from '@/lib/hooks/useServerTableState';

const { sort, dir, page, limit, setSort, setPage, setPageSize, getSortDirection } =
  useServerTableState({ defaultLimit: DEFAULT_PAGE_SIZE });
// URL: ?sort=name&dir=asc&page=2&limit=20
```

> `useServerTableState` re-exports `PAGE_SIZE_OPTIONS`, `DEFAULT_PAGE_SIZE`, `MAX_PAGE_SIZE` from `@/lib/db/utils/pagination` so client + server share constants. Pair with server-side fetch using `@/lib/db/utils/pagination` (see [`sk-db`](../sk-db/SKILL.md)).

**Sort cycle:** Header click cycles `null` (↕️) → `asc` (↑) → `desc` (↓) → back to `null`.

**Cascading filters (SK.md §3.1 — mandatory with 2+ filters):** Each filter computes its options from the subset already filtered by the OTHER filters. For the portable rule and rationale → [`kb-ui`](../kb-ui/SKILL.md).

## 2. Form System — `@/components/form` (NOT shadcn)

> 🔴 **This repo does NOT ship shadcn's `form.tsx`.** There is no `@/components/ui/form`. Use the kit at `@/components/form/*` — it wraps `react-hook-form` + Zod with a zero-boilerplate API.

### 2.1 `Form` + `useForm`

```ts
import { Form, useForm } from '@/components/form';
// src/components/form/Form.tsx — passes `isSubmitting` down via context

interface UseFormOptions<T extends FieldValues> {
  schema: z.ZodType<T>; // Zod schema — NO resolver needed
  defaultValues?: DefaultValues<T>;
  mode?: 'onBlur' | 'onChange' | 'onSubmit' | 'all'; // Default: 'onBlur'
}

interface FormProps<T extends FieldValues> {
  form: UseFormReturn<T>;
  onSubmit: (data: T) => Promise<void> | void;
  children: React.ReactNode;
  className?: string;
}
```

```tsx
'use client';
import { z } from 'zod';
import { Form, FormField, SubmitButton, useForm } from '@/components/form';
import { toast } from 'sonner';

const schema = z.object({
  email: z.string().email('Email inválido'),
  name: z.string().min(2, 'Mínimo 2 caracteres'),
});
type FormData = z.infer<typeof schema>;

export function UserForm({ onSubmit }: { onSubmit: (data: FormData) => Promise<void> }) {
  const form = useForm({ schema, defaultValues: { email: '', name: '' } });

  return (
    <Form
      form={form}
      onSubmit={async (data) => {
        try {
          await onSubmit(data);
          toast.success('Usuario guardado');
        } catch (err) {
          console.error('UserForm submit failed:', err);
          toast.error('No pudimos guardar. Intenta de nuevo.');
        }
      }}
      className="space-y-4"
    >
      <FormField name="email" label="Email" type="email" autoComplete="email" />
      <FormField name="name" label="Nombre" />
      <SubmitButton>Guardar</SubmitButton>
    </Form>
  );
}
```

**Notes:**

- `useForm` takes `{ schema, defaultValues?, mode? }` — NOT `react-hook-form`'s signature. No `resolver` — Zod is wired internally.
- `FormField` reads `register()` from `FormProvider` via `useFormContext()`. No `control` prop, no render props.
- Server-action integration: call the action inside `onSubmit`, destructure `ActionResult<T>` (see [`sk-api`](../sk-api/SKILL.md) §2), `toast.error(result.error)` on failure.

### 4.5 Popover vs DropdownMenu — decisión

Both are Radix primitives that float content over the page. They are **not interchangeable**. Picking the wrong one leaks bugs into derived projects (transparent panels, doesn't close, conflicts with other portals, broken a11y).

**Decision rule:**

- ¿El usuario navega con flechas para elegir UNA acción/opción? → `DropdownMenu`
- ¿El contenido es estructurado (lista scrolleable + header + footer / form / preview / live data)? → `Popover`

**Ejemplo canónico — `src/components/layout/Header.tsx` shippea los 3 casos juntos:**

| Componente        | Primitiva      | Por qué                                                                                                  |
| ----------------- | -------------- | -------------------------------------------------------------------------------------------------------- |
| Avatar menu       | `DropdownMenu` | Acciones a ejecutar (Editar perfil, Cerrar sesión) — `role="menu"` + flechas + cierre al click correctos |
| Theme toggle      | `DropdownMenu` | Selección entre 3 opciones excluyentes — comportamiento de menú radio                                    |
| Notification bell | `Popover`      | Contenido scrolleable con header/footer + items que abren un Dialog — no es menú                         |

**Por qué la distinción importa:**

- `DropdownMenu` agrega `role="menu"` + `role="menuitem"` y cierra al click por default. Forzarlo en contenido estructurado obliga a `e.preventDefault()` en cada botón inline (mark-as-read, links del footer) y rompe la semántica para screen readers.
- `Popover` es contenido genérico — no asume nada del layout interno. Posicionamiento + portal + outside-click + escape gratis, sin las suposiciones de menú.

**🔴 NO rodar tu propio dropdown con `useState` + `useRef` + click-outside listener.** El bell antiguo lo hacía y generó 3 bugs recurrentes en derivados: transparente cuando un wrapper redefine `--background`, no cerraba bien al abrir el detail Dialog (cualquier click dentro del portal externo disparaba close), y se clipeaba con ancestros `overflow:hidden`. Radix Portal resuelve los tres en una primitiva.

```tsx
import { Popover, PopoverContent, PopoverTrigger } from '@/components/ui/popover';

<Popover open={open} onOpenChange={setOpen}>
  <PopoverTrigger asChild>
    <MyTriggerButton />
  </PopoverTrigger>
  <PopoverContent align="end" sideOffset={8} className="w-80 p-0">
    {/* contenido libre — header, lista, footer, form, etc */}
  </PopoverContent>
</Popover>;
```

> Trigger custom debe ser `forwardRef` con spread de props para que Radix le inyecte `ref`, `aria-expanded`, `data-state` (ver `NotificationBell.tsx` como referencia).

## 8. 🔴 Human IDs in UI — never UUIDs

Users never see UUIDs. Always display the human id (`ORD-2026-0042`, `USR-0001`, `TKT-2026-0123`). Pairs with the dual-ID schema pattern in [`sk-db`](../sk-db/SKILL.md).

| Context             | Display           | Use internally     |
| ------------------- | ----------------- | ------------------ |
| Breadcrumbs         | `ORD-2026-0042`   | uuid (hidden)      |
| Tables              | `orderNumber`     | `id` for queries   |
| URLs                | `/orders/ORD-...` | resolve via lookup |
| Toast / email / PDF | Human id          | —                  |
| Phone support       | Human id          | —                  |

Helper:

```ts
import { generateHumanId } from '@/lib/utils/human-id';

generateHumanId(42, { prefix: 'ORD' }); // → 'ORD-2026-0042'
```

## 9. CSS Variables & Theming (brief)

All theming uses CSS custom properties defined in `src/app/globals.css`. **Never hardcode colors** (CODING.md §5).

Prefer Tailwind semantic classes (`bg-background`, `bg-muted`, `bg-destructive`, `text-foreground`, `text-muted-foreground`, `border-border`, `ring-ring`) over raw color classes.

## 10. Kit-specific anti-patterns

| ❌                                                   | ✅                                                                                   |
| ---------------------------------------------------- | ------------------------------------------------------------------------------------ |
| Creating a component without `INVENTORY.md` check    | Reuse `DataTable`, `ConfirmDialog`, form kit, etc.                                   |
| Editing `src/components/ui/*` (shadcn primitives)    | Wrapper in `shared/`, `common/`, or feature folder (SK.md §3.3)                      |
| Using `@/components/ui/form` (does NOT exist here)   | Use the kit's `@/components/form` instead                                            |
| Writing `resolver: zodResolver(schema)` in `useForm` | Pass `{ schema }` — Zod is wired internally                                          |
| Raw `AlertDialog` when `ConfirmDialog` fits          | Use `ConfirmDialog` — less boilerplate                                               |
| Missing `BreadcrumbSetter` on `[id]` routes          | Add one to resolve the dynamic label                                                 |
| Hardcoding pagination constants                      | Re-export from `@/lib/db/utils/pagination`                                           |
| Showing UUIDs in UI                                  | Human IDs from `@/lib/utils/human-id`                                                |
| Hardcoding colors (hex, Tailwind `slate-*`)          | CSS variables / semantic classes (`bg-background`, `text-foreground`)                |
| Building custom delete switch + dialog               | `StatusToggle` from `@/components/shared/StatusToggle`                               |
| Duplicating cascading-filter logic                   | Apply SK.md §3.1 using filter subset computations (see [`kb-ui`](../kb-ui/SKILL.md)) |
