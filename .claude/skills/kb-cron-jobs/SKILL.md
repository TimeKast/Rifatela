# kb-cron-jobs — Portable Cron-Job Patterns for Next.js + Vercel

> Pair: [`kb-ssot-registries`](../kb-ssot-registries/SKILL.md) — meta-pattern this skill applies. Read it first if unfamiliar with the reconcile-vs-validate dichotomy and the static handler check.

**What this skill covers:** the end-to-end recipe for cron jobs in a Next.js app — declarative layer (registry + autogen `vercel.json` + linter), runtime layer (auth, logging, retry, cleanup), operational layer (failure notifications, watchdog, manual invocation, on/off toggles, idempotency). Both audited derivatives independently rebuilt parts of this; this skill is the union plus the lessons from where each one drifted.

> **Naming convention used throughout this skill:** helpers like `startJobRun`, `runWithRetry`, `cleanupStaleLogs` are **patterns the implementer builds in their own project**, not functions the kit base exports today. The skill describes the recommended shape and signature; your project owns the implementation, typically under `src/lib/cron/` or equivalent.

---

## 1. When to use crons — decision tree

| Need                                                            | Pick                                                  |
| --------------------------------------------------------------- | ----------------------------------------------------- |
| Deterministic schedule + low concurrency + idempotent           | **Cron**                                              |
| High concurrency / fan-out / retry-on-failure as first-class    | **Queue** (Inngest, QStash, BullMQ, Trigger.dev, etc) |
| Server interval / always-on loop                                | Persistent server only — never on Vercel/serverless   |
| Limited duration + no DB-heavy work + edge-locality matters     | Edge scheduled function                               |
| User-triggered "run this now" + may take >max function duration | Queue with progress tracking, not cron                |

If your job is "every N minutes/hours/days, do X to the database", cron is the right tool. The rest of this skill assumes that.

## 2. Vercel deployment context

- **Always check current plan/runtime limits before implementing.** Vercel adjusts ceilings periodically and the answer depends on Hobby vs Pro vs Enterprise AND whether Fluid Compute is enabled. Reference: Vercel Cron Jobs pricing page + Vercel functions duration page.
- **Variables that matter for any cron design** — look these up against your current plan + runtime config:
  - Per-project cron count limit
  - Minimum schedule granularity (Hobby is historically more restrictive than Pro)
  - `maxDuration` ceiling — different on Standard runtime vs Fluid Compute runtime
  - Cost model per invocation
- **Long-running jobs** (ERP syncs, batch imports, heavy recomputes) generally require Pro + Fluid Compute. Verify the current ceiling supports your worst-case duration before scheduling.
- **Cron auth.** Vercel auto-injects `Authorization: Bearer ${CRON_SECRET}` into every cron invocation. **Validate it on every cron route** — without the check, the URL is publicly invocable.
- **`export const dynamic = 'force-dynamic'`** mandatory in every cron route. Cron routes must NOT be cached, pre-rendered, or treated as static.
- **`export const maxDuration = N`** in the route handler. **Precedence:** Vercel resolves `maxDuration` from code-level `export` first, then `vercel.json`, then dashboard, then default. The registry is the contract; the linter (§4) cross-checks that the route's exported value matches the registry's `maxDuration` field.

## 3. Apply `kb-ssot-registries` to the cron domain

A single registry, `src/config/cron-jobs.ts`, is the only place a human edits to add, remove, rename, or reschedule a job. Everything else (Vercel config, types, admin UI rows, watchdog cadence) is derived.

Recommended shape (the synthesis of what both audited derivatives do well, with the runtime knobs lifted into the declarative layer):

```ts
// src/config/cron-jobs.ts
export interface CronJob {
  readonly id: string; // snake_case, unique
  readonly path: string; // /api/cron/...
  readonly schedule: string | readonly string[]; // UTC cron, can fan out
  readonly handler: string; // @/lib/jobs/foo#runFooJob (statically checked)
  readonly description: string; // operational docs
  readonly displayName: string; // UI label
  readonly scheduleLabel?: string; // UI override (timezone-friendly)
  readonly category: 'sync' | 'cleanup' | 'compute' | 'health' | 'export';
  readonly maxDuration?: number; // seconds — must match route's export
  readonly retry?: { count: number; delayMs: number };
  readonly notifyOnFailure?: boolean; // default true
}

export const CRON_JOBS = [
  {
    id: 'cron_foo',
    path: '/api/cron/foo',
    schedule: '0 * * * *',
    handler: '@/lib/jobs/foo#runFooJob',
    description: 'Hourly foo recomputation',
    displayName: 'Foo Recompute',
    category: 'compute',
    maxDuration: 60,
  },
] as const satisfies readonly CronJob[];

export type JobId = (typeof CRON_JOBS)[number]['id'];
export const CRON_SCHEDULES = CRON_JOBS.map((j) => ({
  id: j.id,
  displayName: j.displayName,
  scheduleLabel:
    j.scheduleLabel ?? (Array.isArray(j.schedule) ? j.schedule.join(', ') : j.schedule),
  category: j.category,
}));
```

**Lifting the runtime knobs.** A real-world derivative we audited keeps `maxDuration` per-route inline (`export const maxDuration = N` in each `route.ts`, where N is whatever ceiling makes sense for that handler under their Vercel plan + Fluid Compute config). Lifting it to the registry centralizes the contract and lets the linter cross-check the route against the registry. Same for `retry` and `notifyOnFailure` — declarative beats scattered.

## 4. Build pipeline (Vercel deployment recipe)

Three scripts under `scripts/tools/`:

- **`build-vercel-crons.ts`** — pure function `buildVercelJson(): { crons: { path; schedule }[] }`. Iterates `CRON_JOBS`; for each entry, expands `schedule` (if array) into multiple Vercel cron entries with the same `path`. Output is deterministic.
- **`generate-vercel-crons.ts`** — wrapper that calls `buildVercelJson()`, serializes to JSON with stable formatting, compares against the on-disk `vercel.json`, writes only if it differs (idempotent). **No marker field, no marker comment** — `vercel.json` is parsed by Vercel as strict JSON; it must contain only keys Vercel accepts. Drift defense lives in invariants `vercel-sync` (#9) and `vercel-shape` (#12), not in a self-describing payload.
- **`cron-jobs-lint/index.ts`** — validator. Invariants checked:
  1. **`id-format`** — matches `^[a-z][a-z0-9_]+$`
  2. **`id-unique`** — no duplicates
  3. **`path-prefix`** — starts with `/api/cron/`
  4. **`path-route-exists`** — file `src/app${path}/route.ts` exists on disk
  5. **`path-unique`** within registry (`vercel.json` may legally have dups for fan-out schedules)
  6. **`cron-format`** — exactly 5 whitespace-separated fields per schedule string
  7. **`handler-format`** — matches `^@/(lib|app)/.+#[A-Za-z_][\w$]*$`
  8. **`handler-static`** — resolves alias + reads file + regex-matches the named export (see [`kb-ssot-registries`](../kb-ssot-registries/SKILL.md) §4 for the best-effort scope and required actionable error messages)
  9. **`vercel-sync`** — deep-equals the result of `buildVercelJson()` against the on-disk `vercel.json`. Catches manual edits to the derivative.
  10. **`route-orphans`** — walks `src/app/api/cron/`, flags any `route.ts` whose path is not declared in the registry.
  11. **`max-duration-sync`** (recommended) — parses each route's `export const maxDuration` and compares against the registry's `maxDuration` field. Catches drift in the precedence chain (§2).
  12. **`vercel-shape`** — the on-disk `vercel.json` MUST contain only top-level keys that Vercel recognizes. **Required contract** (skill prescribes; project implements):
      - **Offline.** Runs in pre-commit; **no network calls** at lint time. Pre-commit has to work in CI sandboxes and offline editing — the validator reads a vendored copy, never fetches.
      - **Fail-closed on unknown top-level keys.** Any unrecognized key aborts the commit with an actionable error: offending key name + a pointer to https://vercel.com/docs/project-configuration.
      - **Vendor Vercel's official JSON Schema.** Vercel publishes a machine-readable Draft-4 schema at `https://openapi.vercel.sh/vercel.json` (`additionalProperties: false` at root, ~24 top-level keys including `$schema`, `crons`, `headers`, `redirects`, `rewrites`, `routes`, `cleanUrls`, `trailingSlash`, `github`, `git`, `public`, `regions`, `passiveRegions`, `functions`, `functionFailoverRegions`, `images`, `bulkRedirectsPath`, `wildcard`, `webAnalytics`, `alias`, `name`, `builds`, `build`, `env`). **Vendor a snapshot** into the project (e.g., `scripts/cron-jobs-lint/vercel.schema.json`) and validate `vercel.json` against it with Ajv or any Draft-4 validator. Pin the snapshot date.
      - **Caveat — docs occasionally exceed the schema.** Vercel's project-configuration docs reference top-level keys not always present in the openapi.vercel.sh schema (at time of writing: `buildCommand`, `devCommand`, `installCommand`, `framework`, `outputDirectory`, `ignoreCommand`, `bunVersion`, `fluid`). If your project uses any of these, **supplement** the vendored schema with an explicit allowlist for the gap. The supplemental list lives next to the schema snapshot and updates together.
      - **Update policy.** Refresh the vendored schema and supplemental allowlist in the same PR that adopts a new Vercel feature. The lint failure on the unknown key IS the prompt — by design. Cost: one extra commit per Vercel feature adoption. Benefit: zero broken deploys from typo'd or hallucinated keys.
      - **Lightweight alternative.** A project without an Ajv dependency MAY skip schema validation and enforce only "top-level keys ⊆ vendored set" (union of schema keys + supplemental list). Same fail-closed semantics, no Ajv.
      - **Sanity test.** A `vercel.json` whose root contains `"$schema": "https://openapi.vercel.sh/vercel.json"` MUST pass — `$schema` is a recognized top-level key.

Pre-commit hook integration follows [`kb-ssot-registries`](../kb-ssot-registries/SKILL.md) §5 — phase 3 runs `pnpm generate:vercel-crons` (reconciler), phase 4 runs `pnpm cron-jobs:lint` (validator).

Add to `package.json`:

```json
{
  "scripts": {
    "generate:vercel-crons": "tsx scripts/tools/generate-vercel-crons.ts",
    "cron-jobs:lint": "tsx scripts/tools/cron-jobs-lint/index.ts"
  }
}
```

## 5. Route handler conventions

Every cron route follows the same shape:

```ts
// src/app/api/cron/foo/route.ts
import { NextResponse } from 'next/server';
import { runFooJob } from '@/lib/jobs/foo';
import { startJobRun, completeJobRun, failJobRun } from '@/lib/cron/run-tracking';

export const dynamic = 'force-dynamic';
export const maxDuration = 60; // must match registry

export async function GET(request: Request) {
  const authHeader = request.headers.get('authorization');
  const secret = process.env.CRON_SECRET;
  if (!secret) {
    return NextResponse.json({ error: 'CRON_SECRET not configured' }, { status: 503 });
  }
  if (authHeader !== `Bearer ${secret}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  const runId = await startJobRun('cron_foo', 'cron');
  try {
    const result = await runFooJob();
    await completeJobRun(runId, { recordsProcessed: result.processed });
    return NextResponse.json({ ok: true, runId, ...result });
  } catch (err) {
    const message = err instanceof Error ? err.message : String(err);
    await failJobRun(runId, message);
    return NextResponse.json({ ok: false, error: message }, { status: 500 });
  }
}
```

Three things this snippet enforces:

- **Auth before anything else.** A 503 when the secret is missing makes misconfiguration loud; a 401 on mismatch keeps the URL private.
- **The handler library function (`runFooJob`) is the contract** — both the cron route AND the manual-invocation route (§9) call it. No reimplementation.
- **`startJobRun` / `completeJobRun` / `failJobRun` wrap the work** — never `console.log`-only failures (§14).

## 6. Runtime infrastructure — patterns to implement in your project

These helpers belong in your project (typically `src/lib/cron/`) — not in the kit base. The skill describes the recommended shape; you write the implementation once and reuse across routes.

**Recommended DB schema** (table `automation_job_runs` or your equivalent):

```ts
// src/lib/db/schema/cron-runs.ts (recommended shape)
{
  id: uuid().primaryKey().defaultRandom(),
  jobId: varchar({ length: 50 }).notNull(),     // matches registry id
  trigger: enum(['cron', 'manual']).notNull(),
  status: enum(['running', 'success', 'failure', 'partial', 'timeout', 'skipped']).notNull(),
  startedAt: timestamp().notNull().defaultNow(),
  completedAt: timestamp(),
  durationMs: integer(),
  recordsProcessed: integer(),
  errorMessage: text(),
  metadata: jsonb(),
  // index on (jobId, startedAt desc) for "latest run per job" queries
}
```

**Recommended interfaces:**

```ts
// signatures only; your implementation lives in src/lib/cron/
declare function startJobRun(jobId: JobId, trigger: 'cron' | 'manual'): Promise<string /* runId */>;
declare function completeJobRun(
  runId: string,
  opts?: { recordsProcessed?: number; metadata?: object }
): Promise<void>;
declare function failJobRun(
  runId: string,
  errorMessage: string,
  opts?: { metadata?: object }
): Promise<void>;
declare function runWithRetry<T>(
  fn: () => Promise<T>,
  label: string,
  opts?: { count?: number; delayMs?: number }
): Promise<T>;
declare function cleanupStaleLogs(): Promise<void>;
declare function purgeOldRuns(retentionDays: number): Promise<number>;
```

**Two-channel logging.** Every job emits BOTH:

- A structured `logger.info` / `logger.error` line for runtime introspection (Vercel logs, Sentry breadcrumbs).
- A row in the run-tracking table for cross-time queryable history.

They serve different audiences (live debug vs historical audit). `logger.error` alone is not a notification path — see §7.

**Stale log defense.** Functions can crash mid-run before they call `completeJobRun`/`failJobRun`. Without cleanup, the `running` row sits forever. `cleanupStaleLogs()` runs at the **start** of every cron route (piggyback), marks rows older than a threshold (e.g. 15 min) as `timeout`. Idempotent and cheap.

**Retention.** `purgeOldRuns(retentionDays)` is a separate cron. Retention is project-runtime-config (env var like `CRON_LOG_RETENTION_DAYS`) — pick a value matching your audit/compliance window. Apply a defensive size cap on `metadata` jsonb at insert time (e.g. 16 KB) to avoid runaway rows from accidental large dumps.

## 7. Failure detection and notifications

Failure handling is layered:

- **Per-run failure** (handler threw, `failJobRun` ran). Opt-in subscribers receive a notification via the kit's notification system (`sk-notifications` `notify()` API). Strict subscription via category preference, e.g. `'cron_alerts'` — **never broadcast to all admins**.
- **Notification payload.** Title with job displayName + status, body with truncated error message (cap to a sensible length), URL deep-linking to the admin cron panel for context.
- **Misses vs failures.** Both are covered by the watchdog (§8):
  - _Failure_ = last run for jobId completed with `status='failure'`.
  - _Miss_ = no run for jobId within the expected cadence window. Tolerance suggested 1.5× to 2× the schedule period.
- **Logger gotcha.** `logger.error` from inside the cron handler reaches Vercel logs but does NOT trigger user-facing notifications by itself — the DB write + subscription is what fans out the alert. Never assume Vercel logs are someone's primary channel.

The anti-pattern is broadcasting failures to all admins or relying on Vercel email digests. Both lead to alert fatigue → muted alerts → real failures hidden.

## 8. Watchdog meta-cron (recommended for any production cron system)

A separate cron that runs less frequently than the others (e.g. once a day, early morning), queries the run-tracking table, and per `jobId` flags:

- **Misses** — no row in the expected window
- **Failures** — last row has `status='failure'`
- **Stale runners** — rows stuck `running` past `cleanupStaleLogs()`'s threshold (should be 0 if cleanup is wired correctly; non-zero means cleanup itself is broken)

It sends **one consolidated notification per subscriber** covering all flagged jobs (one digest > N spam notifications).

The pattern is meta-monitoring: the watchdog treats the registry as data and asks "did each `jobId` run within its expected cadence?" That question can only be answered by a system that knows the registry, the schedule, and the run history — exactly what the SSOT pattern gives you.

**Computing expected cadence.** To know when a job is overdue, the watchdog parses cron expressions and computes "next run after timestamp X". Use a maintained library (e.g. `cron-parser`) — do not roll your own parser. Edge cases (DST transitions, fan-out arrays, nth-day-of-month) are non-trivial.

## 9. Manual invocation from admin UI

Operators sometimes need to run a cron immediately — to retry after a fix, to backfill a window, to validate a change before the next scheduled run.

Pattern:

- Admin route `POST /api/admin/cron/run/[jobId]` invokes the **same handler library function** the scheduled route does. Logs `startJobRun(jobId, 'manual')` so the run is distinguishable in history.
- **Auth via `withAuth({ resource: 'cron', action: 'run' }, ...)` — NOT `CRON_SECRET`.** The user is authenticated through the app, not arriving from Vercel's cron infrastructure.
- **UI:** "Run now" button on each row of the cron admin panel. Disable while a `running` row exists for that jobId — concurrent runs violate idempotency assumptions (§10).

The anti-pattern is reimplementing the job logic inline in the manual route. Drift between scheduled and manual behavior is silent and dangerous; the handler library function is the contract.

## 10. Idempotency contract

Every cron job MUST be safe to run twice in a row with the same input. Three reasons:

1. Vercel may retry on transient failures.
2. Manual invocation can collide with the scheduled run.
3. The function may restart mid-run after a crash (zombie row scenario, §6).

Patterns that give you idempotency:

- **`INSERT ... ON CONFLICT DO UPDATE`** (`onConflictDoUpdate` in Drizzle) for upserts.
- **Dedupe by external/source id** — e.g. an `external_invoice_id` from an ERP, a hash of the source record.
- **Process-only-items-older-than-X** + **mark-processed-on-success** — re-runs become no-ops because the items are already marked.
- **Watermark patterns** — `processedUpTo: timestamp` per jobId. A re-run resumes from where the last run left off, doesn't duplicate.

Anti-patterns that produce dupes on retry:

- `INSERT` without `ON CONFLICT` — re-run dupes rows.
- Position-based processing ("process the first 10") without a marker — re-run reprocesses the same 10.
- Side effects (email, webhook, payment) without a sent-log dedupe — re-run sends duplicate notifications.

## 11. Runtime on/off toggles

Business sometimes needs to **pause** a cron temporarily — during maintenance, a bad-data window, a migration in progress. Removing the cron from the registry to "disable" it conflates two different things (intent vs state); use a runtime flag instead.

Pattern:

- **`enabled` flag in DB** — one row per jobId in a `cron_settings` table, or a generic feature-flag table. The default is `enabled: true`.
- **Route handler short-circuits** at the start if `enabled === false`. It still calls `startJobRun(jobId, trigger)` + `completeJobRun(runId, { status: 'skipped' })` so the history records the no-op (audit trail preserved).
- **Vercel constraint.** `vercel.json` is locked at deploy time; Vercel will still hit the URL on schedule. The DB-flag short-circuit is what actually pauses the work.
- **Admin UI.** Per-row toggle. Disabling shows a confirmation _"Skipping the next N scheduled runs (until re-enabled)"_ with the next-run timestamp computed from the schedule (`cron-parser`, §8).
- **Stale-disabled warning.** If a job has been disabled for >24h, surface a banner _"Cron disabled since DATE — re-enable or remove from registry"_. Disabled-and-forgotten is a worse failure mode than "broken and noisy" — at least the latter is visible.

The anti-pattern is removing the cron from the registry to disable it. Registry edits should reflect intent ("this job no longer exists"), not state ("this job is paused right now"). Mixing the two corrupts the registry's meaning.

## 12. Admin dashboard derivation

The admin cron panel renders one row per registry entry. Always derive the list from `CRON_SCHEDULES`; never hardcode.

Per-row columns:

- `displayName`
- `scheduleLabel`
- Last-run `status` + `startedAt` + `durationMs`
- `recordsProcessed`
- Retry count (from last run's metadata)
- On/off toggle (§11)
- "Run now" button (§9)
- History link (paginated list of past runs for this jobId)

The anti-pattern observed in an audited derivative: an admin endpoint hardcoding a subset of `JobId` values. The registry has N jobs, the endpoint queries M < N, the panel renders null for the rest. Symptom: rows go silent for weeks before anyone notices. The endpoint must derive its query list from the registry, not from a parallel const.

## 13. Other deploy targets (when you migrate off Vercel)

- **Render** — `render.yaml` jobs. Generate from the registry.
- **Railway** — cron triggers in `railway.toml`. Generate from the registry.
- **Self-hosted** — BullMQ / Inngest / `node-cron`. Registry → schedule registration on boot.

The registry shape is portable; only the generator changes. The runtime patterns (CRON_SECRET-equivalent auth, run-tracking DB, `runWithRetry`, `cleanupStaleLogs`, watchdog, manual-invocation route, on/off toggles, idempotency) carry over unchanged. Numeric constants — `maxDuration` ceiling, retention days — become target-specific.

## 14. Anti-patterns

- **Hand-editing `vercel.json`.** Registry says X, vercel.json says Y; reconciler at the next commit overwrites which is good, but the temptation persists. Reconciler in §4 overwrites on every commit; invariants 9 (`vercel-sync`) + 12 (`vercel-shape`) detect drift in CI.
- **Marker field on derivative (`_generated`, `_meta`, `_factory`) on top-level `vercel.json`.** JSON has no comment syntax; the temptation is to inject a custom top-level field for provenance. **Do not.** Vercel rejects unknown top-level keys at deploy time with no useful error in the GitHub deployment status — diagnosis requires `vercel inspect <deployment>`, which is gated by org access and CLI auth. Provenance lives in: the generator's source code (JSDoc on `buildVercelJson()`), the SSOT registry (`src/config/cron-jobs.ts`), the linter's `vercel-sync` + `vercel-shape` invariants, and git history. Never the derivative.
- **Hardcoding `JobId` strings as literals** in routes/handlers/admin — no derivation from registry. Rename a job → 5 unrelated places break in unrelated PRs.
- **Hardcoded subset in the admin endpoint** (e.g. monitoring 2 of 5 — registry has more).
- **Routes orphan in `src/app/api/cron/*` not in the registry** — `route-orphans` check (§4) catches.
- **Missing `CRON_SECRET` validation** — URL becomes a public footgun.
- **Missing `force-dynamic`** — Next.js may pre-render or cache.
- **Persisting failures only to `console.log`** — no retention, no audit, no monitoring, no notifications.
- **`logger.error` only, no DB row** — Vercel logs are not someone's primary alert channel; alerts come from the DB + subscription.
- **Retry without backoff or limit** — easy to write infinite loops.
- **Watchdog broadcasts to all admins** — alert fatigue.
- **`maxDuration` in route only, not in registry** — drift between declared time-budget and actual handler timeout.
- **Manual-invocation route reimplements job logic** — must call the same handler library function the scheduled route does.
- **Disabling a job by removing it from the registry** — registry encodes intent, not runtime state. Use the `enabled` flag (§11).
- **Non-idempotent jobs** — INSERT without ON CONFLICT, position-based processing without watermark, side effects without dedupe.
