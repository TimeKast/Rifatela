## 1. Barrel layout (what the kit ships)

| Import                           | From                                 | Role                                                    |
| -------------------------------- | ------------------------------------ | ------------------------------------------------------- |
| `withAuth`, `withSelf`           | `@/lib/actions/helpers`              | Action wrappers — RBAC + Zod + revalidate orchestration |
| `ActionResult<T>`, `ActionError` | `@/lib/actions/types`                | Discriminated-union return + user-facing throw          |
| `auth`, `signIn`, `signOut`      | `@/lib/auth`                         | NextAuth v5 barrel (re-exports from `auth.ts`)          |
| `requirePermission`              | `@/lib/auth/permissions`             | Throws when role lacks `(resource, action)` grant       |
| `hasPermission`                  | `@/lib/auth/permissions`             | Non-throwing check (for conditional UI / branch logic)  |
| `logger`                         | `@/lib/logger`                       | Structured server-side logger (do NOT `console.log`)    |
| `canHardDeleteUser`              | `@/lib/db/helpers/can-hard-delete`   | Typed FK-aware eligibility for hard-delete              |

> If any import above is missing in a derived project, the kit's action/route patterns will not compose. Surface the gap rather than reinventing.

## 2. Canonical wrappers — signatures + contract

Both wrappers are the **kit's canonical way** to write a Server Action mutation. They compose the pipeline, parse input, inject `userId`, apply `revalidatePath` on success, and normalize errors into `ActionResult<T>`. Their implementation lives in `@/lib/actions/helpers` (source of truth — reference it, don't re-derive the pipeline).

### 2.1 `withAuth()` — admin / RBAC-gated

Use for actions that require an RBAC permission (admin / CRUD on shared resources). Pipeline: **auth → permission → Zod → handler → revalidate**.

```ts
// Signature (from @/lib/actions/helpers)
export async function withAuth<TInput, TOutput = void>(
  options: {
    resource: Resource; // RBAC resource (see @/lib/auth/permissions)
    action: Action; // RBAC action ('create' | 'update' | 'delete' | …)
    schema: ZodSchema<TInput>; // validated automatically — handler gets TInput
    revalidate?: string; // path revalidated only on success
  },
  input: unknown, // FormData or plain object — wrapper normalizes
  handler: (parsed: TInput, userId: string) => Promise<TOutput>
): Promise<ActionResult<TOutput>>;
```

**Minimal example:**

```ts
// src/lib/actions/admin/user-admin.ts
'use server';

import { eq } from 'drizzle-orm';
import { db } from '@/lib/db/drizzle';
import { users } from '@/lib/db/schema';
import { withAuth } from '@/lib/actions/helpers';
import { ActionError, type ActionResult } from '@/lib/actions/types';
import { createUserSchema } from '@/lib/validations/admin/user-admin';

export async function createUser(input: unknown): Promise<ActionResult> {
  return await withAuth(
    {
      resource: 'users',
      action: 'create',
      schema: createUserSchema,
      revalidate: '/settings/users',
    },
    input,
    async (data, userId) => {
      const existing = await db
        .select({ id: users.id })
        .from(users)
        .where(eq(users.email, data.email))
        .limit(1);

      if (existing.length > 0) {
        throw new ActionError('Ya existe un usuario con ese email');
      }

      await db.insert(users).values({
        ...data,
        createdBy: userId,
        modifiedBy: userId,
      });
    }
  );
}
```

**Contract (what the wrapper guarantees):**

| Step                | Behavior                                                                                                                                         |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Auth check          | `auth()` → missing session returns `{ error: 'Debes iniciar sesión' }` (never reaches handler)                                                   |
| Permission check    | `requirePermission(role, resource, action)` → denial returns `{ error: 'No tienes permiso para <action> en <resource>' }`                        |
| Input normalization | `FormData` → `Object.fromEntries`; plain object passed through. Both run through `schema.safeParse`                                              |
| Schema failure      | First Zod issue message → `{ error: <firstError.message \|\| 'Datos inválidos'> }`                                                               |
| Handler call        | `handler(parsedData, session.user.id)` — `data` is already `TInput` (fully typed, no re-parse needed)                                            |
| Revalidation        | `revalidatePath(options.revalidate)` fires only if handler resolved (not on throw). For tag-based revalidation call `revalidateTag()` in-handler |
| Error mapping       | `ActionError` → `{ error: err.message }` (user-visible). Any other throw → `console.error('[withAuth:<resource>:<action>]', err)` + generic msg  |

**Rules (kit-specific):**

- `'use server'` at the top. Never mix actions and plain utilities in the same file (the whole file becomes RPC surface).
- Input arrives as `unknown`. `withAuth` parses it. Do not Zod-parse again inside the handler — the handler signature already gives you `TInput`.
- Throw `ActionError('<mensaje>')` for anything the user should see. Any other throw → generic fallback (`'Ocurrió un error. Intenta de nuevo.'`) + server log.
- `revalidate: '<path>'` is applied ONLY on success.
- `createdBy` / `modifiedBy` come from the `userId` the wrapper injects — never from the input payload.

### 2.2 `withSelf()` — self-service (no RBAC)

Use `withSelf` when the user is acting on their own data (profile, password, notification prefs). No RBAC check — just auth + schema + revalidate. `withSelf` has **two overloads**:

```ts
// Overload 1 — with schema (validated input)
export async function withSelf<TInput, TOutput = void>(
  options: {
    schema: ZodSchema<TInput>;
    revalidate?: string;
  },
  input: unknown,
  handler: (parsed: TInput, userId: string) => Promise<TOutput>
): Promise<ActionResult<TOutput>>;

// Overload 2 — no schema (auth-only, handler gets userId + email)
export async function withSelf<TOutput = void>(
  options: { revalidate?: string },
  handler: (userId: string, email: string) => Promise<TOutput>
): Promise<ActionResult<TOutput>>;
```

**Validated example:**

```ts
// src/lib/actions/profile.ts
'use server';

import { eq } from 'drizzle-orm';
import { db } from '@/lib/db/drizzle';
import { users } from '@/lib/db/schema';
import { profileSchema } from '@/lib/validations/profile';
import { withSelf } from '@/lib/actions/helpers';
import { type ActionResult } from '@/lib/actions/types';

export async function updateProfile(formData: FormData): Promise<ActionResult> {
  return await withSelf(
    { schema: profileSchema, revalidate: '/profile' },
    formData,
    async (data, userId) => {
      await db
        .update(users)
        .set({ name: data.name, modifiedAt: new Date(), modifiedBy: userId })
        .where(eq(users.id, userId));
    }
  );
}
```

**Auth-only example** (no input — e.g. `sendResetEmail`, `getCurrentProfile`):

```ts
return await withSelf({ revalidate: '/profile' }, async (userId, email) => {
  return getProfileUser(userId);
});
```

> 🔴 Anti-pattern: using `withAuth({ resource, action })` for self-service. That bypasses the "it's my own data" semantic and forces an RBAC grant the user may not have (e.g. a staff user editing their own profile would need `users:update`, which usually only admin has). Use `withSelf`.

## 3. `ActionResult<T>` + `ActionError` — types contract

From `@/lib/actions/types` (importable from both server and client — the file is NOT `'use server'`):

```ts
// Discriminated union — every server action returns this shape
export type ActionResult<T = void> = { data: T; error?: never } | { error: string; data?: never };

// User-facing error — throw inside a wrapper handler
export class ActionError extends Error {
  constructor(message: string) {
    super(message);
    this.name = 'ActionError';
  }
}
```

### When to throw `ActionError` vs return

| Situation                                                                      | Do this                                                        |
| ------------------------------------------------------------------------------ | -------------------------------------------------------------- |
| Business rule violation with a specific user message (uniqueness, FK, quota)   | `throw new ActionError('Ya existe un usuario con ese email')`  |
| Unexpected failure (DB down, timeout, unknown)                                 | Let it throw — wrapper logs + returns generic `{ error: '…' }` |
| Permission-like refusal inside handler (e.g. "can't delete: has dependencies") | `throw new ActionError(eligibility.reason)`                    |
| Success with no payload                                                        | Return nothing — wrapper returns `{ data: undefined }`         |
| Success with payload                                                           | `return value` — wrapper wraps as `{ data: value }`            |

> ⚠️ **Never `return { error: '…' }` directly from a handler.** The handler signature is `(parsed, userId) => Promise<TOutput>` — returning an error object makes `data` the error object. Always `throw new ActionError(...)`.

### On the client

```tsx
const result = await createUser(formData);
if (result.error) {
  toast.error(result.error); // user-facing message
} else {
  // result.data is now TypeScript-narrowed to TOutput
  toast.success('Created!');
}
```

## 4. Zod validation context (wrapper-parsed, not handler-parsed)

The wrapper runs `schema.safeParse(raw)` for you — the handler receives **already-validated `TInput`**. This is different from the portable `ActionResult` pattern in [`kb-api`](../kb-api/SKILL.md), where the action itself calls `safeParse`.

```ts
// ❌ DON'T: re-parse inside the handler
async (data, userId) => {
  const reparsed = createUserSchema.safeParse(data); // redundant, data is already TInput
  if (!reparsed.success) throw new ActionError('Datos inválidos');
  await db.insert(users).values(reparsed.data);
};

// ✅ DO: trust `data` — it's the validated TInput
async (data, userId) => {
  await db.insert(users).values({ ...data, createdBy: userId });
};
```

**FormData vs plain object:** the wrapper normalizes via `input instanceof FormData ? Object.fromEntries(input) : input`. Your schema must therefore describe the **flat** shape (string fields after `FormData` coercion), not the raw `FormData`. For nested objects (e.g. JSON submitted from a client action), pass a plain object — the wrapper forwards it as-is.

**Error message surfacing:** when validation fails, the wrapper returns `{ error: firstError.message }` using `parsed.error.issues[0].message`. Write user-friendly messages in `.min()/.email()/.refine(...)` calls — they surface as the toast.

## 5. CRUD compositions — signatures consumed by `sk-crud-scaffold`

The CRUD scaffolder orchestrates seven canonical actions per entity. Each one composes the wrappers above with a consistent RBAC `(resource, action)` pair and a `revalidate` path that matches the list/detail route. This is the compositional contract the scaffolder generates against — diverge at your peril.

**File:** `src/lib/actions/admin/{entity}.ts` (one file per entity, `'use server'` at top).

| Action                      | Wrapper    | Options                                                                       | Signature                                       | Notes                                                                                   |
| --------------------------- | ---------- | ----------------------------------------------------------------------------- | ----------------------------------------------- | --------------------------------------------------------------------------------------- |
| `get{Entities}()`           | manual     | `auth()` + `requirePermission(role, '{entities}', 'list')`                    | `() => Promise<Row[]>`                          | Read — no wrapper. Filter with `notDeleted({table})` (see [`sk-db`](../sk-db/SKILL.md)) |
| `get{Entity}ById(id)`       | manual     | `auth()` + `requirePermission(role, '{entities}', 'read')`                    | `(id: string) => Promise<Row \| null>`          | Return `null` on miss; caller handles 404                                               |
| `getAdjacent{Entities}(id)` | manual     | `auth()` + `requirePermission(role, '{entities}', 'list')`                    | `(id: string) => Promise<AdjacentPayload>`      | CTE pattern — single query, never multi-query JS Date round-trip                        |
| `create{Entity}(input)`     | `withAuth` | `{ resource: '{entities}', action: 'create', schema, revalidate: '/{list}' }` | `(input: unknown) => Promise<ActionResult>`     | Uniqueness checks via `ActionError`; audit via injected `userId`                        |
| `update{Entity}(id, input)` | `withAuth` | `{ resource: '{entities}', action: 'update', schema, revalidate: '/{list}' }` | `(id, input: unknown) => Promise<ActionResult>` | Find target → guard → update. Bind `id` outside wrapper or inside schema                |
| `delete{Entity}(id)`        | `withAuth` | `{ resource: '{entities}', action: 'delete', schema: idSchema, revalidate }`  | `(id: string) => Promise<ActionResult>`         | Soft delete by default: set `deletedAt` + `deletedBy`. Hard delete → see §6             |
| `restore{Entity}(id)`       | `withAuth` | `{ resource: '{entities}', action: 'update', schema: idSchema, revalidate }`  | `(id: string) => Promise<ActionResult>`         | Clear `deletedAt` + `deletedBy`. Note `'update'` action, not `'restore'`                |

> ⚠️ **CTE mandatory for `getAdjacent{Entities}`** — use a single query, never multi-step with a JS intermediate.
>
> **Root cause:** JavaScript `Date` has **millisecond** precision; PostgreSQL `timestamptz` has **microsecond**. If you fetch `createdAt` from the current row, round-trip it through a `Date` object, and then use it as a bound in a second query (`WHERE createdAt > $prevCreatedAt`), the sub-millisecond component is silently truncated — and rows created in the same millisecond flip sides of the comparison.
>
> **Symptom:** intermittent navigator failures — `getPrev` / `getNext` skip a row or return the same row twice, only reproducible when inserts are dense.
>
> **Fix:** keep the timestamp comparison inside Postgres with a CTE:
>
> ```sql
> WITH current AS (SELECT created_at, id FROM entities WHERE id = $1)
> SELECT prev.id AS prev, next.id AS next
> FROM current
> LEFT JOIN LATERAL (
>   SELECT id FROM entities
>   WHERE (created_at, id) < (current.created_at, current.id)
>   ORDER BY created_at DESC, id DESC LIMIT 1
> ) prev ON true
> LEFT JOIN LATERAL (
>   SELECT id FROM entities
>   WHERE (created_at, id) > (current.created_at, current.id)
>   ORDER BY created_at ASC, id ASC LIMIT 1
> ) next ON true;
> ```
>
> Tuple comparison `(created_at, id)` breaks ties deterministically at the same microsecond. Never materialize the cursor timestamp in JS.

### Revalidate composition

| List route        | Detail route           | What to revalidate                                                                                                     |
| ----------------- | ---------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `/settings/users` | `/settings/users/[id]` | `revalidate: '/settings/users'` covers both (list page re-fetches; detail re-fetches on next nav)                      |
| Tag-based (rare)  | —                      | Omit `revalidate` option; call `revalidateTag('users')` inside the handler and tag the fetch with `{ next: { tags } }` |
| Multi-path (rare) | —                      | Omit option; inside handler call `revalidatePath('/a')` + `revalidatePath('/b')`                                       |

### Pattern: update with id outside the payload

`updateEntity(id, input)` takes two args because the `id` usually comes from the URL (server component param), not the form. Two idiomatic shapes:

```ts
// Option A — id merged into input before calling the wrapper
export async function updateEntity(id: string, input: unknown): Promise<ActionResult> {
  return await withAuth(
    {
      resource: 'entities',
      action: 'update',
      schema: updateEntitySchemaWithId,
      revalidate: '/entities',
    },
    { ...((input as object) ?? {}), id },
    async (data, userId) => {
      // data.id is present and validated
      await db
        .update(entities)
        .set({ ...data, modifiedBy: userId })
        .where(eq(entities.id, data.id));
    }
  );
}

// Option B — id stays outside; schema describes the patch only
export async function updateEntity(id: string, input: unknown): Promise<ActionResult> {
  return await withAuth(
    { resource: 'entities', action: 'update', schema: patchEntitySchema, revalidate: '/entities' },
    input,
    async (patch, userId) => {
      await db
        .update(entities)
        .set({ ...patch, modifiedBy: userId })
        .where(eq(entities.id, id));
    }
  );
}
```

> `sk-crud-scaffold` defaults to Option B (cleaner separation of concerns; schema describes only what the user edits).

Reference: [`sk-crud-scaffold`](../sk-crud-scaffold/SKILL.md) for the full scaffold contract.

## 6. Read-only / listing actions — manual pattern

Read actions that only need auth + a permission check (no mutation, no input) skip the wrapper and call `auth()` + `requirePermission()` directly:

```ts
'use server';

import { auth } from '@/lib/auth';
import { requirePermission } from '@/lib/auth/permissions';
import { redirect } from 'next/navigation';

export async function getUsers(): Promise<UserListItem[]> {
  const session = await auth();
  if (!session?.user?.id) redirect('/login');

  requirePermission(session.user.role, 'users', 'list');

  return await db.select({…}).from(users).orderBy(asc(users.createdAt));
}
```

> `requirePermission` **throws** when denied — do not wrap in try/catch here; the server-side redirect/error boundary takes over. Use `hasPermission` for non-throwing checks.

## 7. Delete-with-dependencies — `canHardDeleteUser`

Soft delete is safe-by-default. Hard delete requires an eligibility check via the typed helper — do NOT invent an ad-hoc `hasMovements`:

```ts
import { canHardDeleteUser } from '@/lib/db/helpers/can-hard-delete';
import { ActionError } from '@/lib/actions/types';

async (data, userId) => {
  const eligibility = await canHardDeleteUser(data.id);
  if (!eligibility.canDelete) {
    throw new ActionError(eligibility.reason ?? 'No se puede eliminar: tiene registros asociados');
  }
  await db.delete(users).where(eq(users.id, data.id));
};
```

For entities beyond `users`, mirror the pattern — one typed helper per table that counts references in specific FK columns (see [`sk-db`](../sk-db/SKILL.md) for the helper shape). Don't generalize prematurely.

## 8. HumanId generation — retry on unique conflict

When inserting a row with a `humanId` backed by a PG SEQUENCE (see [`sk-db`](../sk-db/SKILL.md) human-id section), the sequence can drift after branch restore or seed. Wrap the insert in a retry loop on PG error `23505` for the humanId unique constraint:

```ts
const MAX_RETRIES = 5;

for (let attempt = 0; attempt < MAX_RETRIES; attempt++) {
  const humanId = await getNextHumanId(db, 'user_human_id_seq', {
    prefix: HUMAN_ID_PREFIXES.USER,
    includeYear: false,
  });

  try {
    await db.insert(users).values({ humanId, ...rest });
    return; // success
  } catch (err: unknown) {
    const pgErr = err as {
      code?: string;
      constraint?: string;
      cause?: { code?: string; constraint?: string };
    };
    const code = pgErr.code ?? pgErr.cause?.code;
    const constraint = pgErr.constraint ?? pgErr.cause?.constraint;

    if (code === '23505' && constraint === 'users_human_id_unique') continue;
    throw err; // unrelated error — rethrow
  }
}

throw new ActionError(`No se pudo generar un ID único después de ${MAX_RETRIES} intentos.`);
```

Drizzle wraps the PG error, so check both top-level and `cause.code` / `cause.constraint`. Reference implementation: `src/lib/actions/admin/user-admin.ts → createUser`.

## 9. Route handlers — inline pattern (no central `handleApiError`)

**Kit decision:** this repo does NOT ship an `AppError` / `ERROR_CODES` / `handleApiError` module. Route handlers do inline validation, return `Response.json({ error, message, details? }, { status })`, and log via `@/lib/logger`. The kit's rule: "three similar handlers < one premature abstraction" — abstract only when a real third case appears.

### POST with auth + Zod

```ts
// src/app/api/push/subscribe/route.ts
import { auth } from '@/lib/auth';
import { logger } from '@/lib/logger';
import { subscribePush } from '@/lib/notifications/push';
import { z } from 'zod';

const subscribeSchema = z.object({
  endpoint: z.string().url(),
  keys: z.object({ p256dh: z.string().min(1), auth: z.string().min(1) }),
});

export async function POST(req: Request) {
  const session = await auth();
  if (!session?.user?.id) {
    return Response.json({ error: 'Unauthorized' }, { status: 401 });
  }

  try {
    const body = await req.json();
    const result = subscribeSchema.safeParse(body);

    if (!result.success) {
      return Response.json(
        { error: 'Invalid request body', details: result.error.flatten() },
        { status: 400 }
      );
    }

    await subscribePush(session.user.id, result.data, req.headers.get('user-agent') ?? undefined);
    return Response.json({ ok: true }, { status: 201 });
  } catch (error) {
    logger.error('[api:push:subscribe] POST failed', { error });
    return Response.json({ error: 'Internal server error' }, { status: 500 });
  }
}
```

### Dynamic params (Next.js 16+ — async)

```ts
// src/app/api/avatar/[userId]/route.ts
const paramsSchema = z.object({ userId: z.string().uuid() });

export async function GET(_req: Request, { params }: { params: Promise<{ userId: string }> }) {
  const { userId } = paramsSchema.parse(await params);
  // …
}
```

> Next.js 16+: `params` is a Promise — `await` before parsing. Same for `searchParams` in Server Components.

### Public endpoint (no auth)

Invite validation, health, NextAuth callbacks — no `auth()` call. Still validate input with Zod/`safeParse` and wrap in try/catch.

### Error shape convention (kit-canonical)

```ts
{ error: 'snake_case_code', message: 'User-facing text', details?: { … } }
```

Common codes seen in the repo: `Unauthorized`, `invalid_request`, `invalid_token`, `user_exists`, `server_error`, `Invalid request body`, `Internal server error`. Keep codes stable — they're read by clients.

> Reference files: `src/app/api/health/route.ts` (200/503 degraded), `src/app/api/invites/accept/route.ts` (validated POST), `src/app/api/push/subscribe/route.ts` (auth + Zod).

## 10. Webhooks (when you add one)

The repo doesn't ship any provider webhooks yet. When adding one:

1. Verify signature **before** reading the payload (raw body is required — don't `req.json()` first).
2. Keep handlers idempotent — key on the provider's event id.
3. Log the event type at info level; never log the full payload.

```ts
export async function POST(request: Request) {
  const body = await request.text(); // raw body for signature check
  const signature = request.headers.get('x-provider-signature');
  if (!verifySignature(body, signature)) {
    return Response.json({ error: 'Invalid signature' }, { status: 400 });
  }
  const event = JSON.parse(body);
  // handle…
  return Response.json({ received: true });
}
```

## 11. Anti-patterns (kit-specific)

| ❌                                                                                                 | ✅                                                                                       |
| -------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Inventing `AppError` / `ERROR_CODES` / `handleApiError`                                            | `ActionError` + `ActionResult` for actions; inline `Response.json` for routes            |
| Hand-rolling `auth()` + `requirePermission()` + `safeParse()` + `revalidatePath()` in every action | `withAuth` / `withSelf` from `@/lib/actions/helpers` — the wrapper exists for this       |
| Using `withAuth({ resource, action })` for self-service                                            | `withSelf` — `withAuth` forces an RBAC grant the user may not have on their own data     |
| Zod-parsing a second time inside the handler                                                       | Trust the parsed `data` the wrapper passes in — handler sig already types it as `TInput` |
| `return { error: '…' }` from inside the handler                                                    | `throw new ActionError('…')` — returning makes the error the `data` payload              |
| `createdBy` / `modifiedBy` from input payload                                                      | From the `userId` the wrapper injects                                                    |
| Inventing a new `hasMovements` per table                                                           | `canHardDeleteUser` (or analogous typed helper from `sk-db`)                             |
| Throwing `Error('unauthorized')` inside an action                                                  | Let `withAuth` return the standard message — don't bypass                                |
| Mixing actions and helpers in one `'use server'` file                                              | Helpers go in a non-`'use server'` file like `actions/types.ts`                          |
| `console.log` for observability                                                                    | `@/lib/logger` (structured)                                                              |
| Generic `catch { return 500 }` without log                                                         | `logger.error('[scope] action failed', { error })` then return 500                       |

- [ ] Delegates to `withAuth` (RBAC) or `withSelf` (self)
- [ ] Zod schema lives in `@/lib/validations/**`, passed via options
- [ ] User-facing errors thrown as `ActionError`
- [ ] `revalidate` path or tag set in options (or explicit `revalidateTag` inside)
- [ ] `createdBy` / `modifiedBy` set from `userId` param

**Server Action (read):**

- [ ] `await auth()` + `redirect('/login')` or `ActionError` on missing session
- [ ] `requirePermission(role, resource, action)` called (or documented why skipped)

**Route Handler:**

- [ ] `safeParse` for body AND params (`await params` in Next.js 16+)
- [ ] Explicit status code on every response
- [ ] Errors logged via `@/lib/logger`, not `console`
- [ ] `auth()` check when the endpoint is private
- [ ] Response shape: `{ error, message, details? }` on failure

- [ ] Idempotency keyed on provider event id
